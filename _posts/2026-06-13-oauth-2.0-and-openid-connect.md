## 1. Why this document exists

The SSO flow involves a lot of moving pieces — `state`, `nonce`, PKCE verifier, code challenge, signed cookies, two trips to the IdP, JWTs, redirects.
This doc walks through the *why* behind each piece.

---

## 2. The core problem SSO solves

You are building an app. Users need to log in.

You **could** ask every user for a username and password, store hashes, handle resets, deal with breaches. That's "local auth" it's what most apps still do.

But your users probably already have a Google account, or a Microsoft Entra account, or an Okta account at their company. Asking them for *another* password means:

- Another password for them to forget.
- Another password database for you to defend.
- Another set of "forgot password" emails for you to maintain.
- A worse user experience.

**SSO ("single sign-on") is the idea: let users prove they are who they say they are using an account they already have, somewhere else.**

The "somewhere else" is called the **Identity Provider** (IdP). Google, Microsoft, Okta, Keycloak, and Auth0 are all IdPs.

The promise of SSO is: *the user types their password into the IdP, never into us.* That's the whole point. If our backend ever sees the user's IdP password, we've lost the security benefit.

---

## 3. OAuth 2.0 — what it is

OAuth 2.0 is a security protocol that lets an application access a user's data on another website without learning that user's password.
It answers the question: "What is this app allowed to do?"

**How It Works**

* **Redirection**: The app sends the user to log in directly on the trusted platform (e.g., Google or GitHub).
* **Token Issuance**: After approval, the platform gives the app an Access Token instead of the password.
* **Limited Access**: This token acts like a temporary key card with strict boundaries, defined by:
    * **Scopes**: Permissions to do specific actions (e.g., read:photos but not delete:account).
   * **Expiries**: A short lifespan (e.g., valid for only 1 hour).

OAuth 2.0 is strictly about authorization (permissions). It does not identify who the user is.

Example request and response 
```bash
# request
https://google.com?
  client_id=my_fitbit_app&
  response_type=code&
  redirect_uri=https://fitbit.com&
  scope=https://googleapis.com

# response
access_token
```

## 4. OpenID Connect (OIDC) — what it adds

**OpenID Connect (OIDC, 2014)** is a thin standardization layer on top of OAuth 2.0. It adds:

1. **The `id_token`** : a signed JWT that contains identity claims (`sub`, `email`, `name`, etc.) directly.
2. **A `nonce` parameter** : defends against replay of stolen id_tokens.
3. **The `openid` scope** : Including it forces the IdP to return an id_token alongside standard OAuth tokens.
4. **A discovery document** : at `https://issuer/.well-known/openid-configuration`, the IdP publishes all its endpoints and public keys. Clients fetch it once at startup and stop hardcoding URLs.
5. **Standardized claims** : every OIDC IdP agrees that `sub` is the stable user ID, `email` is the email, `aud` is the intended audience, etc.

**TL;DR:** OAuth is *"can this app do something for the user?"* OIDC is *"who is the user?"*

Example request and response 
```bash
# request
https://google.com?
  client_id=my_fitbit_app&
  response_type=code&
  redirect_uri=https://fitbit.com&
  scope=openid email profile

# response
{
  "iss": "https://google.com",
  "sub": "9876543210",
  "aud": "my_fitbit_app",
  "exp": 1781358600,
  "email": "john.doe@gmail.com",
  "name": "John Doe"
}

```

## 5. The hostile delivery channel

The hardest thing to internalize about OAuth/OIDC is this:

> **The IdP cannot talk to your backend directly. It can only send messages through the user's browser.**

There's no preexisting trust relationship. Google has never heard of `subdomain.example.com`. The first time anyone at our company creates a Google OAuth client, we get a `client_id` + `client_secret`, but those let *us* call Google — they don't let Google call us.

So when the user logs in at Google and Google has the answer ("yes, this is alice@example.com"), Google's only way to deliver that answer is:

> "Hey browser, redirect to https://subdomain.example.com/callback?code=abc123"

The user's browser walks the answer over to us.

This means the message travels through:

- The user's browser process.
- Browser history.
- Browser extensions.
- Server access logs (ours, any reverse proxy, any CDN in front).
- Whatever sits in between (corporate proxies, NATs, etc.).
- `Referer` headers when the callback page links anywhere.

In other words, **the delivery channel is publicly observable and potentially adversarial**. Every protection in OAuth/OIDC exists because of this fact.

The four protections are:

| Seal | Defends against | Consequence |
|---|---| --- |
| `state` | "An attacker tricks the user into visiting our callback with the attacker's auth code, logging the user in as the attacker." (CSRF) | Activity happens on attacker's acount while user thinks it is his own, data theft can happen |
| `nonce` | "An attacker who somehow obtains a valid id_token replays it on a fresh login flow." (token replay) | Attacker impersonates the legit user, has access to an legit account |
| PKCE | "An attacker who steals the auth code from the URL/logs/Referer redeems it for tokens." (code interception) | Auth code is exchanged for id_token, if we lack PKCE, attacker can obtain a legit id_token hence getting access to user's account and data |
| Registered `redirect_uri` | "An attacker tricks the IdP into delivering the code to a URL they control." (open redirector) | Browser's response uri |

If you remove any one of these, a specific class of attack becomes possible. They are not paranoia — they are the minimum baseline.

## 6. The Authorization Code flow with PKCE

It is the modern recommended flow for *every* OAuth/OIDC app — server-side, mobile, single-page, anything.

### 6.1 Actors

- **User** - a real human in front of a real browser.
- **Browser** - the user's web browser. Holds cookies. Follows redirects.
- **Client** - Application backend. We are the "OAuth client."
- **IdP** - Google, Microsoft, Okta, Keycloak, etc. Also called "Authorization Server."

### 6.2 Setup (one-time, before any user logs in)

When a human admin registers any Client Application with the Identity Provider (IdP), they receive two core pieces of information:

- **`client_id`** - public identifier of the application.
- **`client_secret`** - secret known only to application and the IdP. Used in server-to-server calls.
- **A registered `redirect_uri`** - e.g. `https://subdomain.example.com/api/v1/auth/oidc/callback`. The IdP will refuse to redirect anywhere else.

Application at startup also fetches the IdP's discovery document (`/.well-known/openid-configuration`) which contains:

- `authorization_endpoint` - where to send the user's browser to log in.
- `token_endpoint` - where application calls server-to-server to exchange code for tokens.
- `jwks_uri` - where to fetch the IdP's public signing keys.

### 6.3 The flow, narrated

#### Step 1 - User clicks "Sign in with SSO"
Browser navigates to: `GET subdomain.example.com/api/v1/auth/oidc/login`

#### Step 2 - Application prepares three random secrets
- `state` (32 random bytes, base64url'd)
- `nonce` (32 random bytes)
- `code_verifier` (48 random bytes - the PKCE secret)

Application computes `code_challenge = SHA256(code_verifier)`.

#### Step 3 - Secure the Login Context in a Cookie
The application packages the temporary session variables into a signed cookie named application_oidc_state on the user's browser.

```bash
{
  "state": "random_string",
  "nonce": "random_string",
  "code_verifier": "pkce_secret_string",
  "return_to": "/dashboard",
  "issued_at": 1781358600
}
```

**Security Properties**

- **HMAC-Signed:** Signed with the backend JWT secret to prevent user tampering.
- **HttpOnly:** Blocks JavaScript access to prevent cross-site scripting (XSS) theft.
- **Secure:** Only transmitted over encrypted HTTPS connections.
- **SameSite=Lax:** Allows the browser to send the cookie back when returning from the external IdP redirect.
- **Path=/api/v1/auth/oidc:** Restricts cookie transmission strictly to the auth routes, keeping general API calls lightweight.
- **5-Minute Lifetime:** Automatically expires to minimize the attack window for intercepted traffic.

The HMAC means the user can read the cookie but not modify it without us noticing. Application holds nothing server-side; the cookie carries everything.

#### Step 4 - Application redirects browser to the IdP
The redirect URL contains: `client_id`, `redirect_uri`, `scope=openid email profile`, `state`, `nonce`, `code_challenge`, `code_challenge_method=S256`, `response_type=code`.

Note: `code_verifier` is NOT in the URL. Only the challenge (the hash). The verifier stays in our cookie.

#### Step 5 - User authenticates at the IdP
User types their password (and MFA if applicable) at the IdP's login page. This is the *only* place a password is entered. Application never sees it.

#### Step 6 - IdP redirects browser back to Application
`GET subdomain.example.com/api/v1/auth/oidc/callback?code=abc123&state=<echoed>`

The browser sends the `application_oidc_state` cookie because it's a top-level navigation matching the cookie's path.

#### Step 7 - Application verifies the cookie
Recompute the HMAC. If it doesn't match → reject (tampered). If `IssuedAt + 5min < now` → reject (expired). Otherwise unpack the payload.

#### Step 8 - Application clears the cookie
One-shot. Even if the rest of the callback fails, the cookie is dead.

#### Step 9 - Application checks state matches
URL's `?state=` must equal the cookie's stored state. **CSRF gate.**

#### Step 10 - Application exchanges code for tokens (the second trip to the IdP)
This is a direct backend-to-IdP call. No browser involved.

`POST <token_endpoint>` with: `code`, `client_id`, `client_secret`, `code_verifier`, `redirect_uri`, `grant_type=authorization_code`.

The IdP checks:
- `client_id` + `client_secret` are valid.
- `code` is unused, unexpired, was issued for this `client_id`.
- `SHA256(code_verifier) == code_challenge` from earlier (PKCE).
- `redirect_uri` matches what was used in step 4.

If all pass, the IdP returns:
```json
{
  "access_token": "...",
  "id_token": "eyJ... (a signed JWT)",
  "token_type": "Bearer",
  "expires_in": 3600
}
```

#### Step 11 - Application verifies the id_token
- Fetch the IdP's public keys (cached from `jwks_uri`).
- Verify the JWT signature.
- Check `iss` matches the configured issuer.
- Check `aud` contains our `client_id`.
- Check `exp` is in the future.

#### Step 12 - Application checks nonce matches
The id_token contains a `nonce` claim. It must equal the nonce in the cookie. **id_token replay gate.**

#### Step 13 - Application extracts identity claims
From the id_token: `sub` (stable user ID), `email`, `email_verified`, `preferred_username`.

#### Step 14 - Application issues its own JWT
Application creates a fresh JWT. HS256, signed with `JWT_SECRET`, 24h expiry, contains `user_id`, `username`, `role`.

#### Step 15 - Application redirects to UI with token
`302 Location: subdomain.example.com/login/sso-success#token=<jwt>`
The backend issues a 302 redirect to send the user back to the front-end application interface, appending the final JSON Web Token (JWT) inside a URL fragment.

#### Step 16 - UI extracts token
SPA route reads `window.location.hash`, parses out `token`, stashes in localStorage, scrubs the hash from the URL bar, navigates to the user's destination. From here on, every API call sends `Authorization: Bearer <jwt>`.

## 7. The four seals - deep dive

### 7.1 `state` - CSRF protection

**Attack defended:** an attacker tricks the user into visiting our callback URL with the attacker's auth code. Without state, we'd exchange the attacker's code, get the attacker's id_token, and log the user in *as the attacker*. The attacker then reads what the user does in "their" account.

**Mechanism:** Application generates a random state when starting the login, stores it in the cookie, sends it to the IdP. The IdP echoes it back in the callback URL. We compare URL's state vs. cookie's state. Mismatch → reject.

**Where it lives in transit:** URL parameter on both legs (login redirect and callback).

**Specs:** required by OAuth 2.0 (RFC 6749 §10.12).

### 7.2 `nonce` - id_token replay protection

**Attack defended:** an attacker somehow obtains a valid, recently-issued id_token (leaked from a server log, compromised server, intercepted on a misconfigured proxy). Without nonce, they could present this id_token to our callback and log in as the original user.

**Mechanism:** Application generates a random nonce, stores it in the cookie, sends it to the IdP. The IdP **embeds** it inside the id_token (which is signed). We extract the nonce from the id_token claims and compare to the cookie. Mismatch → reject.

**Why this works where state doesn't:** state lives in URL parameters - anyone can put any string in a URL. Nonce lives *inside the signed token* - an attacker can't change it without invalidating the IdP's signature.

**Where it lives in transit:** URL parameter on the way out (login → IdP); inside the id_token on the way back.

**Specs:** OIDC Core §3.1.2.1.

### 7.3 PKCE - code interception protection

**Attack defended:** the auth `code` travels through the browser. If an attacker intercepts it (browser history, log files, malicious extension, MITM on a misconfigured network), they could redeem it for tokens.

**Mechanism:**
1. Application generates `code_verifier` (random 48 bytes → 64 chars).
2. Application computes `code_challenge = BASE64URL(SHA256(verifier))`.
3. Application sends `code_challenge` to the IdP (in the URL, where it might leak).
4. Application keeps `code_verifier` in the cookie (never in the URL).
5. When application redeems the code, it sends `code_verifier`.
6. IdP checks `SHA256(verifier) == challenge`. If match, returns tokens; if not, rejects.

An attacker who steals only the code can't redeem it because they don't know the verifier (it never traveled through the browser).

**Why it works:** SHA256 is one-way. Knowing the challenge doesn't let you guess the verifier.

**Where it lives in transit:** challenge in URL (login → IdP); verifier in direct backend call (Application → IdP).

**Specs:** RFC 7636.

### 7.4 Registered `redirect_uri` — open redirector defense

**Attack defended:** an attacker crafts a login URL with `redirect_uri=https://attacker.com/steal` and tricks the IdP into delivering the auth code to the attacker's site.

**Mechanism:** when application is registered with the IdP, an admin pre-declares the allowed redirect URIs (e.g. only `https://subdomain.example.com/api/v1/auth/oidc/callback`). The IdP enforces this — it will not redirect to any URI not on the allowlist.

**Specs:** OAuth 2.0 §3.1.2.

### 7.5 Why `state` and `nonce` aren't merged

A reasonable question: both are random strings, both are checked, why not use one value for both?

**Answer:** functionally you can. Many implementations do (or derive nonce from state via hashing). The reason an application keeps them separate:

- State leaks more (it's in URLs, logs, history). Keeping nonce out of those exposed channels is free.
- They're checked at different times (state immediately, nonce after token exchange) — separate values give clearer audit logs.
- Different specs (OAuth vs. OIDC), so library code paths often handle them differently.
- Generating two random strings costs nothing.

It's hygiene, not a hard requirement.

## 8. Token taxonomy

An application deals with five distinct credentials. Each does one thing.

| Token | Issued by | Lifetime | Where it lives | Who reads it |
|---|---|---|---|---|
| `state` cookie | Application | 5 min | HttpOnly cookie scoped to `/api/v1/auth/oidc` | Application callback handler |
| Authorization `code` | IdP | seconds, single-use | URL query param (briefly) | Application callback handler |
| `id_token` | IdP | minutes | Application memory only, never persisted | Application callback handler |
| `access_token` | IdP | hours | Application memory only, never persisted | used to call IdP APIs |
| Session JWT | Application | `JWT_EXPIRY` | UI localStorage → `Authorization: Bearer` | every application API |

A few subtleties:

- **The state cookie carries the verifier and nonce too.** It's not just for `state`; it's the full scratchpad for the in-flight login.
- **The auth `code` is single-use.** Even if you steal it, you only get one shot, and only if you also have the PKCE verifier.
- **The `id_token` is what we *actually* care about.** It contains the user's identity directly.
- **The `access_token` is for calling IdP APIs.** OIDC returns it because OAuth always returns it.
- **The session JWT is ours, not the IdP's.** From the moment we issue it, the IdP is no longer involved.

## **TL;DR**

OAuth 2.0 lets an application obtain permission to access resources on behalf of a user **without ever seeing the user's password**. OpenID Connect (OIDC) builds on OAuth and adds **identity**, allowing the application to know **who the user is** via a signed `id_token`.

In the modern Authorization Code + PKCE flow:

1. The application generates a `state`, `nonce`, and PKCE `code_verifier`.
2. The user is redirected to the Identity Provider (Google, Microsoft, Okta, etc.) and authenticates there.
3. The IdP redirects the browser back with a short-lived authorization `code`.
4. The application exchanges that code for tokens using the PKCE verifier.
5. The application validates the signed `id_token`, creates its own session, and logs the user in.

Four security mechanisms protect the flow:

* **`state`** → prevents CSRF/login injection attacks.
* **`nonce`** → prevents replay of stolen `id_token`s.
* **PKCE** → prevents stolen authorization codes from being redeemed.
* **Registered `redirect_uri`** → prevents tokens from being delivered to attacker-controlled sites.

The key idea is that **all messages from the IdP must travel through the user's browser**, which is considered an untrusted delivery channel. OAuth and OIDC exist largely to make that browser-mediated exchange secure.
